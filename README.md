# x-dock
Это утилита для создания docker окружения разработчика под определённое приложение. Подключаемся по ssh или VSCode remote и работаем.

Зачем нужно docker окружение для разработчика?
- Все разработчики пользуются одним окружением независимо от настроек домашней ОС.
- При любой смене ОС, или ПК, или рабочего места, всегда можно воссоздать рабочее окружение одной командой.
- Все пути файлов заранее известны.
- Всё заранее настроено из коробки.
- Изолированная среда.

# Инструкция

## Создание своего my-app-dock
Эта часть является ядром и предполагается, что будет включаться в каждый my-app-dock как submodule
Вместо x в имени "x-dock" подставляется имя вашего приложения.

- Создаём папку и в ней свой git репозиторий `git init`
- `git submodule add https://github.com/AIG-Livny/x-dock.git common`
- Добавляем файлы как показано в папке `example`.
- Изменяем config как нравится
- Делаем `./dock install` и в системе появится команда `my-app-dock`

Список функций выводится командой `my-app-dock help`

## Первое использование

Впервый раз нужно создать образ `my-app-dock makeimg`, либо скопировать его у коллеги. После создания образа можно работать.
Перейдите в папку с приложением и введите `my-app-dock ins`. ins это часть вашего кода, функция которая запустит окружение (ins - instance)

## Повседневное использование

- `my-app-dock ins` откроет терминал в свеже запущенном контейнере
- Либо подключаемся к серверу VSCode
- `my-app-dock clean` удалит все контейнеры принадлежащие `my-app-dock`. Либо `my-app-dock kill [id]` для удаления конкретного

## Разработка
Если приходится часто пересобирать образ, то можно использовать apt-cacher-ng, это кеширующий прокси для пакетов убунты. На хосте
```
sudo apt install apt-cacher-ng
```
Все автоматически поднимется, а x-dock его подхватит во время выполнения следующего apt install внутри контейнера. Посмотреть что закешировано можно в /var/cache/apt-cacher-ng
